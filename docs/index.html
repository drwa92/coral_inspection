This page accompanies the paper "LLM-Guided Adaptive Mission Planning for Autonomous Underwater Vehicles in Coral Reef Monitoring". It presents the system overview, architecture, semantic action interface, simulation setup, and a practical guide for using the coral_inspection ROS package with a BlueROV2 in Gazebo.

Abstract

Coral reef monitoring missions are long-horizon, multi-stage, and must operate under uncertain underwater conditions. Traditional waypoint-based planners lack the ability to adapt online, reason over mission context, or provide interpretable decisions for human supervisors. This work proposes an LLM-guided adaptive mission planning framework for Autonomous Underwater Vehicles (AUVs) applied to coral reef monitoring. A constrained semantic action interface, coupled with a persistent mission memory and event-driven replanning, allows natural-language mission instructions to be translated into executable, geometry-aware survey plans. The system is implemented on a BlueROV2 platform in a Gazebo-based UUV simulation, demonstrating robust multi-site coral inspection, memory-aware continuation, and operator-interpretable autonomy.

System Overview

The framework introduces a structured Large Language Model (LLM)-guided mission planning layer on top of a conventional underwater robotics stack. Instead of hand-crafting waypoint sequences, operators describe high-level coral monitoring objectives in natural language, and the system produces safe, interpretable, and geometry-aware mission plans that can be executed by a BlueROV2.

LLM-Guided Planning

Natural-language instructions are translated into structured JSON mission plans using a constrained LLM planner.

Mission Memory

Persistent memory tracks visited sites, completed actions, and events to support long-horizon missions.

Event-Driven Replanning

Execution or safety events trigger replanning using updated mission memory and environment context.

Human-in-the-Loop

Operators can inspect plans, monitor execution, and intervene when needed through ROS topics and tools.

System Architecture

The system is organized into four main ROS nodes, all launched by a single mission configuration file in the coral_inspection package.

Core Components

  • LLM Planner (llm_planner_node.py) – Converts user prompts into JSON mission plans using mission memory and constraints.
  • Mission Memory (coral_mission_memory.py) – Maintains a structured representation of mission history, sites, and events.
  • Event Monitor (coral_event_monitor.py) – Observes events and triggers replanning when conditions are met.
  • Action Executor (coral_action_executor.py) – Executes each semantic action and publishes execution status and memory events.

High-Level Data Flow

Natural-Language Prompt → LLM Planner → Semantic Plan (JSON) → Action Executor → Vehicle Control

During execution, events and state are fed into the Mission Memory. When significant events occur, the Event Monitor triggers replanning, and the LLM Planner produces an updated mission plan grounded in the updated memory.

<launch>
  <node pkg="coral_inspection" type="llm_planner_node.py"
        name="llm_planner" output="screen">
    <param name="user_prompt_topic" value="/coral_captain/user_prompt"/>
    <param name="plan_topic"        value="/coral_captain/plan"/>
    <param name="status_topic"      value="/coral_captain/llm_status"/>
    <param name="openai_config"
           value="$(find coral_inspection)/config/openai_keys.yaml"/>
    <param name="memory_service"
           value="/coral_captain/get_memory"/>
  </node>

  <node pkg="coral_inspection" type="coral_mission_memory.py"
        name="coral_mission_memory" output="screen">
    <param name="event_topic"        value="/coral_captain/memory_event"/>
    <param name="state_topic"        value="/coral_captain/memory_state"/>
    <param name="get_memory_service" value="/coral_captain/get_memory"/>
  </node>

  <node pkg="coral_inspection" type="coral_event_monitor.py"
        name="coral_event_monitor" output="screen">
    <!-- Replanning logic, optional parameters -->
  </node>

  <node pkg="coral_inspection" type="coral_action_executor.py"
        name="coral_action_executor" output="screen">
    <param name="vehicle_ns"      value="/bluerov2"/>
    <param name="plan_topic"      value="/coral_captain/plan"/>
    <param name="status_topic"    value="/coral_captain/executor_status"/>
    <param name="wait_for_motion" value="true"/>

    <param name="sites_yaml"
           value="$(find coral_inspection)/config/sites.yaml"/>
  </node>
</launch>

Semantic Action Interface

The LLM is constrained to a predefined set of semantic actions to guarantee safety and executability. Plans are represented as an ordered list of actions with parameters:

Action Categories

  • Navigation: go_to, go_to_site, waypoint_list, waypoint_file
  • Survey: survey_site, survey_rectangle, survey_circle_rings
  • Inspection: circular, helical
  • Perception: hover_observe, take_photo
  • Control: hold, station_keep
  • Mission/Safety: return_home, replan, emergency_stop

Example Plan

{
  "Plan": [
    {"action": "survey_rectangle", "site": "A"},
    {"action": "hold"},
    {"action": "go_to_site", "site": "B"},
    {"action": "survey_circle_rings", "site": "B"},
    {"action": "hold"},
    {"action": "return_home"},
    {"action": "hold"}
  ]
}

Simulation Environment

All experiments are conducted in a simulated underwater environment using Gazebo and UUV Simulator with a BlueROV2 model. The environment includes multiple coral sites with different geometries and depths.

  • Simulator: Gazebo + UUV Simulator
  • Robot: BlueROV2 (UUV Simulator model)
  • Sensors: Depth, odometry, forward and downward cameras
  • Sites: Rectangular and circular coral sites with configurable survey patterns

The sites.yaml configuration defines site positions and extents, which are used by semantic survey actions such as survey_rectangle and survey_circle_rings.

Getting Started

1. Installation

Follow the installation instructions in the GitHub README to:

  • Install ROS Noetic and Gazebo
  • Install UUV Simulator
  • Clone and build coral_inspection in your catkin_ws

2. Launch the Simulation

cd ~/catkin_ws
source devel/setup.bash

roslaunch coral_inspection coral_llm_demo.launch

This will start Gazebo, spawn the BlueROV2, load the underwater environment, and start the LLM planner, mission memory, event monitor, and action executor nodes.

3. Submit a Mission Prompt

The LLM planner listens to /coral_captain/user_prompt:

rostopic pub /coral_captain/user_prompt std_msgs/String \
"data: 'Survey site A using a lawnmower pattern, then inspect site B with an orbit and return home.'"

4. Inspect the Generated Plan

rostopic echo /coral_captain/plan

5. Monitor Execution

# Executor status
rostopic echo /coral_captain/executor_status

# LLM planner status
rostopic echo /coral_captain/llm_status

# Mission memory state
rostopic echo /coral_captain/memory_state

Runtime Topics & Monitoring

Key Topics

  • /coral_captain/user_prompt – Natural-language user mission instructions
  • /coral_captain/plan – JSON semantic mission plan
  • /coral_captain/llm_status – Planner status and diagnostics
  • /coral_captain/executor_status – Execution progress and outcomes
  • /coral_captain/memory_event – Events that affect mission memory
  • /coral_captain/memory_state – Current summarized mission memory
  • /bluerov2/pose_gt – Ground-truth pose of the BlueROV2

List All Coral Captain Topics

rostopic list | grep coral_captain

Triggering the Executor Directly (Testing)

You can manually publish a minimal plan to test the executor:

rostopic pub /coral_captain/plan std_msgs/String \
"data: '{\"Plan\": [{\"action\": \"hold\"}]}'"
